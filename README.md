## Инструкция по запуску
1. Сформировать файл конфигурации из шаблона `configs/config.example.yaml` и сохранить в `configs/config.yaml`
2. Сформировать файл .env из шаблона `.env.example`
3. Добавить публичный сертификат от ElasticSearch в файл `configs/elastic.pem`
4. Запуск сервиса в docker `docker-compose up -d --build`

## Архитектура и логика работы сервиса

### Общая схема

Сервис реализует асинхронную обработку логов с последующей агрегацией метрик и генерацией алертов.
![Архитектура](docs/images/architecture.png)

---

## Обработка логов

### Pipeline обработки

1. Логи поступают от внешних сервисов батчами в очередь **Kafka**
2. Далее логи обрабатываются модулем обработки логов, проходя следующие этапы:

* валидация входных данных
* сохранение в **Elasticsearch**
* обработка meta-параметров
* расчет метрик и сохранение в **Redis**
* проверка и генерация алертов

### Параллелизм обработки

Следующие этапы выполняются асинхронно (в отдельных goroutine):

* сохранение в Elasticsearch
* обработка meta-переменных
* расчет метрик
* обработка алертов

Однако:

* этап **обработки алертов** выполняется *синхронно после расчета метрик*, так как имеет от него зависимость

---

## Обработка meta-параметров

Meta-параметры — это дополнительные поля, передаваемые вместе с логом (например: `request_id`, `user_id` и др.).

Особенности реализации:

* схема meta-параметров (имя и тип) хранится в **PostgreSQL**
* используется для:

  * динамического формирования фильтров
  * поддержки поиска в пользовательском интерфейсе

---

## Расчет метрик

Метрики рассчитываются в процессе обработки логов и инкрементально обновляются в **Redis**.

### Поддерживаемые метрики

1. Количество запросов по кодам ответа (по сервису)
2. Общее количество запросов (по сервису)
3. Количество логов по уровням:

   * `ERROR`
   * `WARNING`
   * `INFO`
   * `CRITICAL`
4. Количество уникальных пользователей:

   * за час
   * за сутки

---

## Обработка алертов

### Конфигурация алертов

Правила алертов хранятся в таблице `alert_rule` и включают:

* `level` — уровень логов
* `threshold` — порог срабатывания
* `message` — текст уведомления
* `interval` — период агрегации (в минутах)
* `user_id` — получатель уведомления

### Особенности обработки

* алерты вычисляются на основе агрегированных метрик
* выполнение строго после этапа расчета метрик

---

## Отправка алертов

Процесс отправки:

1. Публикация события алерта в **Kafka**
2. Проверка в БД на дублирование алертов
3. Отправка уведомления (в Telegram)
4. Сохранение информации об отправленном алерте в БД

## Фоновые задачи

### Агрегация метрик

Агрегация выполняется по принципу **скользящего окна**:

```
start_time = now - period_time - lag
end_time   = now - lag
```

где:

* `now` — текущее время
* `period_time` — период агрегации (5 сек, 1 мин, 10 мин, 1 час, 1 день)
* `lag` — фиксированная задержка (10 секунд)

Назначение `lag` — компенсация задержек доставки логов.

### Планировщик

* базовая агрегация (5 секунд) запускается каждые 5 секунд
* при кратности времени:

  * 1 минуте → запускается агрегация за 1 минуту
  * 10 минутам → за 10 минут
  * 1 часу → за 1 час

---

### Очистка метрик

#### Краткоживущие метрики

* метрики с шагом 5 секунд:

  * хранятся только в **Redis**
  * TTL — 2 минуты

#### Долгосрочные метрики (в БД)

| Период   | Время хранения |
| -------- | -------------- |
| 1 минута | 1 день         |
| 10 минут | 7 дней         |
| 1 час    | 1 месяц        |
| 1 день   | 6 месяцев      |

#### Очистка

* выполняется ежедневно в 00:00
* удаляются данные, превышающие TTL

## Web API

Сервис предоставляет HTTP API со следующими возможностями:

1. Аутентификация и авторизация пользователей
2. Полнотекстовый поиск по логам (Elasticsearch)
3. Фильтрация по meta-параметрам
4. Получение агрегированных метрик
5. Стриминг метрик в реальном времени (WebSocket)
